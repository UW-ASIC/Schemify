The reader implements tag-dispatch line parsing per the RESEARCH Pattern 1. Structure:

```zig
const std = @import("std");
const types = @import("types.zig");
const props_mod = @import("props.zig");
const root = @import("root.zig");

const Schematic = root.Schematic;
const ParseError = types.ParseError;
// ... import all element types from types ...

/// Parse an XSchem .sch or .sym file from raw bytes into a Schematic.
/// All allocations go into the Schematic's arena.
/// Returns error on malformed input per D-06.
pub fn parse(backing: std.mem.Allocator, data: []const u8) ParseError!Schematic { ... }
```

Key implementation details:

1. **Line iteration**: Split `data` by `\n`. Track line number for error context. Trim `\r` from line endings.

2. **Tag dispatch** (first character of each line):
   - `'v'` -> skip (version header: `v {xschem version=...}`)
   - `'*'` -> skip (comment)
   - `'L'` -> parseLineShape: parse `L layer x0 y0 x1 y1 {attrs}` -> append to `schematic.lines`
   - `'B'` -> parseRect: parse `B layer x0 y0 x1 y1 {attrs}` -> append to `schematic.rects`. If layer==5, also create a Pin entry (XSchem convention: layer 5 rects are pins).
   - `'P'` -> parsePolygon: parse `P layer npoints x1 y1 ... xn yn {attrs}` -> skip for now (polygons stored as lines or rects depending on point count -- complex, defer geometry to Phase 4)
   - `'A'` -> parseArc: parse `A layer cx cy radius start_angle sweep {attrs}` -> append to `schematic.arcs`
   - `'T'` -> parseText: parse `T {text} x y rot mirror hsize vsize {attrs}` -> append to `schematic.texts`. Text content is brace-delimited and may span multiple lines.
   - `'N'` -> parseWire: parse `N x0 y0 x1 y1 {attrs}` -> append to `schematic.wires`. Extract `lab=` from attrs into `net_name` if present.
   - `'C'` -> parseComponent: parse `C {symbol_ref} x y rot flip {attrs}` -> append to `schematic.instances`. Property block may span multiple lines (track brace depth). Use `props_mod.parseProps` for the property string. Store props in `schematic.props` with `prop_start` and `prop_count` on the Instance.
   - `'K'` -> parseKBlock: parse `K {key=val ...}` -> extract `type`, `format`, `template`, `extra` into `schematic.k_type/k_format/k_template/k_extra`. Set `schematic.file_type = .symbol`.
   - `'G'` -> parseGBlock: parse `G {content}` -> check for `type=` (old XSchem format). Set file_type if found.
   - `'S'`, `'V'` -> skip (SPICE/Verilog global properties -- out of scope per REQUIREMENTS.md)
   - `'E'`, `'F'` -> skip (tEDAx/Spectre -- out of scope)
   - else -> return `error.UnknownElementTag` per D-06

3. **Multi-line handling (CRITICAL -- see RESEARCH Pitfall 1)**: When parsing C (component) or T (text) elements, the property/content block can span multiple lines within `{...}` braces. Track `brace_depth: i32` as a parser state variable:
   - When `brace_depth > 0`, accumulate the current line into a buffer (append newline + line content)
   - When `brace_depth` returns to 0, the block is complete -- process it
   - Do NOT dispatch lines while inside a multi-line block

4. **Field parsing**: Use `std.mem.tokenizeAny(u8, line_after_tag, " \t")` to split fields. Parse floats with `std.fmt.parseFloat(f64, field)`. Parse ints with `std.fmt.parseInt(i32, field, 10)`. On parse failure, return the appropriate `ParseError` variant per D-06.

5. **File type detection (see RESEARCH)**: Default to `.schematic`. Set to `.symbol` when K block is found. Also detect via G block type= (old format).

6. **Pin extraction from B (Rect)**: When a B element has `layer == 5`, extract pin name from attrs `name=` property and create a Pin entry. Pin direction from `dir=` attribute. This is how XSchem stores pins in .sym files.
